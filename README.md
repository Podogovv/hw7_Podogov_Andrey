2 Обосновать стратегию деплоя (развертывания, Blue-Green, Canary, Rolling, Shadow) и оценить влияние на риски

Была выбрана стратегия Blue-Green Deployment

Так как в приложении отсутствует обработка ошибок, система обладает повышенным риском отказа при обновлении. Rolling Deployment и Canary Deployment требуют механизмов автоматического обнаружения ошибок. В текущих условиях эти стратегии могут привести к распространению дефектной версии среди пользователей.

Blue-Green Deployment минимизирует риски при обновлении ML-сервиса, поскольку новая версия полностью изолирована от production до момента переключения трафика. Это позволяет протестировать новую ML-модель и приложение перед вводом в эксплуатацию. Дополнительно обеспечивается быстрый откат при обнаружении проблем.

Но внедрение Blue-Green Deployment приводит к увеличению инфраструктурных затрат, так как требуется поддерживать два одинаковых окружения, а также повышает сложность администрирования и риск ошибок синхронизации между окружениями. Несмотря на это, преимущества в отказоустойчивости, безопасности релизов и возможности быстрого отката делают данную стратегию наиболее подходящей для ML-сервиса.

3 Реализовать стратегию развертывания

Реализована стратегия развертывания Blue-Green Deployment. Для этого были созданы два независимых окружения: Blue и Green. Окружение Blue содержит стабильную версию приложения, а Green используется для развертывания новой версии. Для каждой версии был создан отдельный файл конфигурации Docker Compose: docker-compose.blue.yml и docker-compose.green.yml. В качестве балансировщика нагрузки используется Nginx, который перенаправляет пользовательский трафик на активное окружение через upstream-конфигурацию.

Перед переключением трафика новая версия запускается в Green-окружении и проходит проверку через endpoint /health. После успешной проверки Nginx перенаправляет весь трафик на новую версию приложения. В случае возникновения ошибок или нестабильной работы новой версии выполняется rollback — трафик мгновенно возвращается на Blue-окружение без повторного деплоя и без downtime для пользователей.

Выбор Blue-Green Deployment обусловлен тем, что в приложении отсутствует обработка ошибок, поэтому риск отказа production-сервиса при обновлении является высоким. Данная стратегия минимизирует влияние ошибок новой версии на пользователей, обеспечивает безопасное тестирование обновлений и позволяет быстро восстановить работоспособность системы при возникновении проблем.

Проверка /health
- Blue
curl http://localhost:5001/health
- Green
curl http://localhost:5002/health

Запуск Blue: 
- docker-compose -f docker-compose.blue.yml up -d

Запуск Green: 
- docker-compose -f docker-compose.green.yml up -d

Запуск Nginx: 
- docker-compose -f docker-compose.nginx.yml up -d

Как работает стратегия:
- Blue = стабильная версия
- Green = новая тестируемая версия
- Nginx управляет направлением трафика
- При успешной проверке /health трафик переключается на Green
- При ошибках выполняется rollback на Blue без downtime

4 Спланировать A/B-тестирование для ML-модели

Код в файле AB-test.py

Описание эксперимента:

* Группа A использует текущую версию модели

* Группа B использует новую версию модели

* Основная метрика CTR (или accuracy/conversion rate)

* Пользователи случайным образом делятся между двумя версиями

* Проверка статистической значимости выполняется при уровне значимости 0.05

* Если p-value < 0.05, новая модель считается более эффективной и может быть переведена в production
